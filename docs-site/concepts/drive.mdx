Before you can assert or judge anything, you have to put the agent to work. **Driving** is the part of `test(t)` where you send input and get a result back — `t.send()`, `t.sendFile()`, `t.newSession()`, and the HITL handshake (`t.respond()` / `t.respondAll()`). Everything you drive produces a **Turn**, and every assertion and judge call in niceeval reads from a Turn's data — see [Assert](/concepts/assert) and [Judge](/concepts/judge) for what happens next.

## `t.send()` and the `Turn` it returns

`t.send(input)` is the single verb. Under the hood it calls the agent adapter's `send(input, ctx)` and normalizes whatever comes back into a `Turn`:

```ts
const turn = await t.send("What's the weather in Brooklyn today?");

turn.message;      // last assistant message, string
turn.data;          // structured output, if the agent returned any
turn.status;        // "completed" | "failed" | "waiting"
turn.events;        // this turn's StreamEvent[]
turn.usage;          // { inputTokens, outputTokens, ... } if the adapter reports it
turn.expectOk();    // throws if status === "failed"; otherwise returns the turn
```

`t.reply` is shorthand for the last assistant message on the primary session — equivalent to `turn.message` from the most recent `t.send()` call. `t.events` is the primary session's accumulated event stream across every turn sent so far.

<Tip>
`turn.expectOk()` is a precondition helper, not a scoring assertion: it throws immediately (aborting the rest of `test(t)`) rather than recording a failed assertion. Use it when a later turn depends on an earlier one succeeding — chaining assertions onto a failed turn usually just produces confusing secondary failures.
</Tip>

## Attaching files — `t.sendFile()`

`t.sendFile(path, text?)` reads a local file (relative to the eval's directory), infers its MIME type from the extension, and attaches it as a data URL to the turn's input:

```ts
const turn = await t.sendFile("fixtures/invoice.png", "What's the total on this invoice?");
t.check(turn.message, includes("$"));
```

## Multi-turn conversations

Every `await t.send(...)` call is a new turn on the same conversation. Assign each turn to a variable so you can assert on it individually, in addition to the run-level assertions on `t`:

```ts
const draft = await t.send("Draft a follow-up email.");
draft.expectOk();
t.check(draft.message, includes("Best,"));
draft.judge.autoevals.closedQA("Is the tone professional?").atLeast(0.6);

await t.send("Looks good, send it.");
t.calledTool("send_email");
```

Multi-turn `t.send()` requires the agent to declare the `conversation` capability — without it, the adapter has no obligation to remember what happened in the previous turn, and a second `t.send()` call may talk to a fresh, unrelated context. See [Agents & Adapters](/concepts/agents-adapters) for the full capability list.

## Independent sessions — `t.newSession()`

`t.newSession()` opens a second, independent conversation thread that runs alongside the primary one. It returns a session handle with the same drive API (`send`, `sendFile`, `respond`, `respondAll`) and the same scoped-assertion vocabulary as `t`, but scoped to only what happened on that session:

```ts
await t.send("My name is Alex.");
await t.send("What's my name?");
t.check(t.reply, includes("Alex"));         // primary session remembers

const fresh = t.newSession();
await fresh.send("What's my name?");
t.check(fresh.reply, satisfies((r) => !r.includes("Alex"), "no memory leak"));
```

<Warning>
A common mistake is assuming `t.newSession()` guarantees isolation. Isolation depends entirely on the adapter reading `ctx.session.isNew` and honoring it — an adapter that ignores `isNew` and always resumes the same underlying context will make `t.newSession()` silently share state, with no error raised. If you're writing the adapter, verify isolation with an eval like the one above before trusting it.
</Warning>

## Human-in-the-loop (HITL)

Some agents stop mid-turn to ask for approval or missing information instead of finishing outright. When that happens, the turn ends with `status: "waiting"` and one or more `input.requested` events describing what's being asked.

```ts
const draft = await t.send("Draft a follow-up email, but hold before sending.");
draft.parked();                              // t.parked() asserts status === "waiting"

const request = t.requireInputRequest({
  prompt: /send it/i,
  optionIds: ["approve", "reject"],
});

await t.respond({ request, optionId: "approve" });
t.calledTool("send_email");
```

`t.requireInputRequest(filter)` turns a pending HITL request into a concrete value you can inspect and respond to — it throws if zero or more than one pending request matches the filter, so fill in as many filter fields as you can (`id`, `prompt`, `display`, `action`, `optionIds`, `input`) to disambiguate. `t.respond(...)` answers it and sends the next turn; behind the scenes it's just another `send` call carrying your answer.

When a turn has several identical pending requests that should all get the same answer (for example, approving a batch of file changes one by one), use `t.respondAll(optionId)` instead of resolving them one at a time:

```ts
await t.send("Submit each of these changes for approval.");
t.requireInputRequest({ display: /approval/i });

await t.respondAll("approve");
t.succeeded();
```

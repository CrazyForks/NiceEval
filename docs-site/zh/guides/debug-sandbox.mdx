沙箱默认在每个 Attempt 结束后销毁，排查依据是落盘的 artifact：`niceeval show` 能看到判定、断言、diff 和事件流。大多数问题到这里就够了。

但有些问题只能进活的环境里看：

- **环境起不来**——setup 阶段装依赖失败、agent CLI 启动不了。这时 agent 还没开始跑，事件流是空的，最快的办法是进沙箱手动重跑一遍安装命令。
- **改动落在 `git diff` 之外**——全局装了什么包、`$HOME` 下写了什么配置、`PATH` 实际是什么，artifact 里没有。
- **重跑太慢**——冷启动加安装要几分钟，想逐条验证猜测时，留着现场比每次重跑快得多。

## 跑的时候保留现场

```bash
npx niceeval exp local onboarding/tool-first --keep-sandbox
```

`--keep-sandbox` 是 `niceeval exp` 的运行参数：判定为 `failed` 或 `errored` 的 Attempt（包括超时打断的）会把沙箱保留下来，其余照常销毁。通过的 Attempt 不保留——现场是用来解释失败的；调试 setup 钩子想拿一个现场看环境，让一条 eval 失败就行。

运行结束后，摘要里会列出保留了哪些沙箱、怎么进去：

```text
Kept sandboxes (1)
  @1x7f3q9k  onboarding/tool-first #1  errored  docker · a3f9c2d1
             enter: docker exec -it a3f9c2d1 bash
Stop them with: niceeval sandbox stop --all
```

每行给三样东西：Attempt 定位符（用 `niceeval show @1x7f3q9k` 看落盘证据）、沙箱实例 id、进入现场的命令。进去之后就是这次 Attempt 跑完时的环境，可以手动执行命令、翻文件、复现失败。

## 查看和清理

保留下来的沙箱逐条记录在 `.niceeval/sandboxes/` 里，用 `niceeval sandbox` 管理：

```bash
niceeval sandbox list              # 列出保留的沙箱和存活状态
niceeval sandbox stop a3f9c2d1     # 销毁指定沙箱（id 可以只写唯一前缀）
niceeval sandbox stop --all        # 全部销毁
```

`stop` 是幂等的：沙箱已经不在了（手动删过、云端过期）不算错误，只会把记录移掉并说明。如果 provider 销毁失败，命令会保留记录并返回错误，方便稍后重试，不会把仍活着的资源从列表里藏掉。忘了清也有提醒——下次运行开始时，如果还有上次保留的沙箱，会打一行提示。

## 各 Provider 的差别

- **Docker**：容器不会自己消失，是唯一需要主动清理的 provider。除了 `niceeval sandbox stop`，也可以用 `docker ps -f label=niceeval.keep-candidate=true` 直接核对。
- **E2B / Vercel Sandbox**：云端微 VM 到自己的会话时限后自动过期，不清理也不会一直计费；`niceeval sandbox list` 会把已过期的标成 `expired`。Vercel 的会话只有几分钟，保留窗口相应短。
- **自定义 Provider**：`defineSandbox` 产出的 provider 不支持留存，因为事后的 `sandbox stop` 不加载用户配置，无法在新进程里安全找回自定义销毁函数。

## 边界

保留的沙箱只用来排查，不能续跑或重新评分；判定、断言、diff 这些结论仍以 artifact 为准。查看 artifact 的方法见[查看结果](/zh/guides/viewing-results)。

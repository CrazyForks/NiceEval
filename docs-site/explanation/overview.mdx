NiceEval's core design separates "the evaluation logic" from "how the subject under test is connected". NiceEval owns discovery, scheduling, scoring, and reporting. Adapters call the system under test. Sandbox Providers isolate the filesystem.

## Four layers

```text
Eval files / fixtures
        ↓
NiceEval
        ↓
Adapter
        ↓
Subject under test / Sandbox Provider
```

## What NiceEval owns

<CardGroup cols={2}>
  <Card title="Eval discovery" icon="magnifying-glass">
    Finds `*.eval.ts` files and fixture directories, then derives stable IDs from paths.
  </Card>
  <Card title="Scheduling" icon="timeline">
    Controls the run pool size, retries, attempts, and early exit.
  </Card>
  <Card title="Assertions and scoring" icon="chart-bar">
    Collects `t.check`, scoped assertions, judge scores, and test results.
  </Card>
  <Card title="Cache" icon="database">
    Skips cases that already passed when the fingerprint of their inputs is unchanged.
  </Card>
  <Card title="Reports" icon="file-lines">
    Writes console, JSON, JUnit, and other reports.
  </Card>
  <Card title="Artifacts" icon="folder-open">
    Saves summaries, event streams, transcripts, diffs, and test output.
  </Card>
</CardGroup>

## Adapter

<Note>
  `Agent` is the abstraction NiceEval sees. `Adapter` is the concrete implementation you write. NiceEval does not know your agent's private protocol, CLI arguments, or auth rules.
</Note>

That boundary keeps NiceEval generic. Evaluating your own AI agent, Claude Code, Codex, or a custom agent does not change the logic of the runner or the scorers.

## Key terms

<AccordionGroup>
  <Accordion title="Eval">
    A test case: a description and a `test(t)` function, agent-neutral — which Agent it runs against is decided by the experiment.
  </Accordion>
  <Accordion title="Agent">
    A connection NiceEval calls by name, responsible for returning a standard `Turn`.
  </Accordion>
  <Accordion title="Adapter">
    The concrete implementation of an `Agent`, which knows how to call your runtime or CLI.
  </Accordion>
  <Accordion title="Sandbox">
    The isolated runtime environment used by coding agents.
  </Accordion>
  <Accordion title="Turn">
    The immutable result snapshot of one `t.send()`.
  </Accordion>
  <Accordion title="Artifact">
    A structured result file written to disk after a run.
  </Accordion>
  <Accordion title="Experiment">
    A run configuration that compares multiple agents, models, or flags as a matrix.
  </Accordion>
</AccordionGroup>

## End-to-end flow

<Steps>
  <Step title="Discovery">Find eval files and fixtures.</Step>
  <Step title="Scheduling">Plan the run from concurrency, cache, and attempts.</Step>
  <Step title="Agent send">Call the adapter so the subject under test produces a `Turn`.</Step>
  <Step title="Scoring">Run assertions, judge checks, and tests.</Step>
  <Step title="Verdict">Fold every result into `passed`, `failed`, `errored`, or `skipped`.</Step>
  <Step title="Reporting & artifacts">Write reports and save structured files.</Step>
</Steps>
